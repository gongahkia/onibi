# Designing a Multi-Transport Approval Protocol for Local AI Agents

AI coding agents are getting good enough to leave running, but not quite good enough to leave unattended. The failure mode is familiar: you start a Claude Code or Codex session, it makes progress, then it pauses at the exact moment you step away. Maybe it wants to delete a directory. Maybe it wants to rewrite a config file. Maybe it wants to run a migration. You are on a train, in the kitchen, or trying to get lunch, and the run is blocked on a yes/no prompt sitting on your laptop.

Onibi is my answer to that awkward middle state. It is a local cockpit for AI coding agents with a phone in your pocket. Agents run in a desktop or headless daemon. When they request permission, Onibi shows the approval on the desktop and sends it to a mobile PWA. The phone view includes the tool, command, working directory, and recent output. You can approve, deny, or edit the payload before approving.

The edit step is the product. A plain notification that says "Claude wants to run rm -rf tests/legacy" is useful, but it still leaves you choosing between a risky allow and a clumsy deny. The better interaction is "change that to mv tests/legacy tests/legacy.bak, then continue." Claude Code's `updatedInput` support makes that possible for the lead adapter. Onibi turns that capability into a protocol and UI that can work across transports and, eventually, across agents.

## Why Not Just SMS?

The obvious shortcut is to send a message to a phone: SMS, Telegram, Pushbullet, Discord, whatever is easiest. That gets a notification delivered, but it does not solve the protocol problem.

The agent hook has to block until a decision exists. The decision must be structured, not a text reply that a shell script tries to parse. The client needs to reconnect and replay pending approvals after network changes. The phone should see the same approval state as the desktop. The daemon must keep running with no SaaS account and no hosted relay. And the approval payload must allow an edited tool input where the agent supports it.

That pushes the design toward a small local protocol:

- `POST /v1/approval/request` for adapters to create a pending request and long-poll for the decision.
- `POST /v1/approval/:id/decide` for desktop or phone clients to resolve it.
- `WS /v1/realtime` for approval-pending, approval-resolved, run-event, and terminal-output messages.
- `GET /v1/qr` for pairing a phone with the machine ID, token, and available transports.

Everything is JSON. Every remote call requires a bearer token. The protocol carries `machine_id`, `session_id`, and `approval_id` from day one, even though v1.5 is mostly a single-machine experience. That is boring plumbing now and a migration avoided later.

## The Pieces

The desktop app is Tauri: Rust for the local server and system integration, React for the cockpit UI, xterm.js for terminal rendering. The same Rust binary can run headless on Linux or a Raspberry Pi, which is important for the "AI lab in a closet, approve from your phone" use case.

The approval store is SQLite. Pending approval requests are persisted, then also held in an in-memory waiter map so an adapter can block on a oneshot response. If the daemon restarts, clients can still fetch pending approvals from disk. If no one answers within the timeout, the adapter receives a deny with reason `timeout`.

The mobile client is a PWA. It is not as deep as a native app, but it has the right launch properties: installable, cross-platform, cheap to ship, and good enough for approval cards, terminal mirror, and Web Push. iOS still requires the PWA to be added to the home screen before push notifications work, so the onboarding says that directly.

## Why Three Transports?

The hardest part of a phone-first local tool is not rendering a button. It is reachability.

When the phone is on the same Wi-Fi, LAN is enough. Onibi generates a self-signed certificate, starts a LAN TLS listener, and publishes a fingerprint in the pairing payload. The annoying part is real: the user has to trust the cert at the OS level. Browsers cannot pin a self-signed cert from JavaScript before TLS succeeds. LAN is useful, but it should not be the only path.

Tailscale Funnel is the best daily-driver transport for people already in the Tailscale world. It gives a stable HTTPS URL under the tailnet DNS name and avoids opening inbound firewall ports. The tradeoff is setup: Tailscale, login, MagicDNS, HTTPS certs, and Funnel policy all have to be enabled.

Cloudflare Quick Tunnel is the fast demo path. `cloudflared tunnel --url http://127.0.0.1:17893` prints a random `trycloudflare.com` URL and starts forwarding. It is ephemeral, but that is acceptable for trials, screenshots, and "try this on your phone right now" moments.

The pairing payload includes every enabled transport. The PWA can try Tailscale, then Cloudflare, then LAN, and reconnect through the next reachable URL when the current one fails. This is the reason Onibi treats transport as a first-class part of the product instead of a docs footnote.

## Security Shape

Onibi is not a sandbox. If you approve a destructive command, the command can still be destructive. The goal is to make the approval boundary explicit, visible, and reachable.

The v1.5 hardening pass adds rate limiting to approval requests, body limits on large JSON endpoints, WebSocket message caps, HSTS for tunnel-bound HTTPS responses, and a CSP for the served PWA. Tokens live in the OS keychain where possible, with a `0600` file fallback for headless Linux and Pi environments. The fallback is documented as a risk, not hidden as magic.

The daemon never evaluates approval JSON. Adapter payloads are validated as protocol envelopes, stored as JSON, displayed as JSON or command text, and returned as structured decisions.

## The Claude Code Dependency

The lead demo depends on Claude Code's ability to accept an edited input from a `PreToolUse` hook. That matters because "approve with edits" should not cost another LLM turn. If the model proposes `rm -rf tests/legacy`, and the user changes it to `mv tests/legacy tests/legacy.bak`, the hook response should carry the edited input and let the agent continue.

Codex is useful but less complete at launch: Bash can be intercepted, but `apply_patch` and MCP tools are not. Other agents are mostly mirrored shell sessions until their blocking hook surfaces are stable enough to support the same pattern. Onibi is honest about that in the adapter matrix because pretending every agent has the same integration surface would make the tool worse.

## What Is Next

v1.6 is about tightening the operational surface: per-device tokens, revocation, an audit log UI, better adapter coverage, and an SSE fallback for networks that dislike WebSockets. v2.0 is where a hosted relay may make sense, but only as an option. The local-first path should stay real.

The code is Apache-2.0. Fork it, ship adapters, argue with the protocol, and open issues for the parts that are too sharp. The category is still early. I want Onibi to be a useful reference implementation for one simple idea: local agents can do more when permission is not trapped on the laptop screen.
