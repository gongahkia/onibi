# Onibi Approval Protocol v1.0

Status: frozen for Onibi v1.5 Phase-03.

## 1. Overview

Onibi is a single Rust binary with two runtime modes:

- GUI mode: launches the Tauri desktop app and the local approval server.
- Headless mode: `onibi --headless [--port N]` launches only the approval server.

Both modes expose the same HTTP and WebSocket protocol. The default bind address is `127.0.0.1:17893`. Phase-03 is loopback-only; later transport phases may publish the same protocol over LAN or tunnel endpoints.

All request and response payloads are JSON unless an endpoint explicitly states otherwise. Server-Sent Events are deferred to v1.1. Every protocol envelope includes `protocol_version: "1.0"`; clients must reject other versions and servers return `426 Upgrade Required` for explicit incompatible versions.

Authentication is a single shared bearer token generated on first boot:

```http
Authorization: Bearer <token>
```

## 2. Identity

Onibi uses ULIDs for stable, sortable identifiers.

- `machine_id`: generated on first boot, persisted locally, and included on every approval, run event, and WebSocket message.
- `session_id`: generated per PTY or agent session.
- `approval_id`: generated per approval request.
- `deviceId`: generated by `/v1/pair`; informational in v1.0 because all devices share the same bearer token.

## 3. Endpoints

### `POST /v1/approval/request`

Adapters post a tool approval request. The server persists it, broadcasts `approval-pending`, then long-polls for a user decision. The default timeout is 600 seconds.

Request:

```json
{
  "protocol_version": "1.0",
  "machine_id": "01J...",
  "session_id": "01J...",
  "agent": "claude-code",
  "tool": "Bash",
  "input": { "command": "rm -rf node_modules" },
  "cwd": "/path/to/workspace",
  "metadata": { "prompt": "delete node_modules" }
}
```

Response:

```json
{
  "protocol_version": "1.0",
  "approval_id": "01J...",
  "decision": "allow",
  "updatedInput": { "command": "echo skipped" },
  "reason": null
}
```

`updatedInput` is `null` or omitted unless the user edited the tool input before approving.

### `GET /v1/approval/pending`

Returns all undecided approvals, ordered by creation time. Clients use this to populate an inbox after launch or reconnect.

### `POST /v1/approval/:id/decide`

Submits a decision.

Request:

```json
{
  "decision": "allow",
  "updatedInput": { "command": "echo skipped" },
  "reason": "edited by desktop"
}
```

Returns `200` with the decision envelope on success. Returns `409 Conflict` if another client already decided or the approval no longer exists.

### `POST /v1/run/event`

Adapters post lifecycle events.

```json
{
  "protocol_version": "1.0",
  "machine_id": "01J...",
  "session_id": "01J...",
  "kind": "started",
  "timestamp": "2026-05-27T01:00:00Z",
  "payload": {}
}
```

`kind` is one of `started`, `tool-use`, `output-chunk`, or `completed` in v1.0. Unknown kinds must be displayed or logged, not rejected, so third-party adapters can experiment.

### `POST /v1/pty/output`

Posts high-volume terminal tail data. `data` is base64-encoded bytes. The server fans out via WebSocket and keeps only an in-memory rolling buffer of the last 5000 bytes per session.

```json
{
  "protocol_version": "1.0",
  "machine_id": "01J...",
  "session_id": "01J...",
  "data": "SGVsbG8K"
}
```

### `GET /v1/qr`

Returns `image/png`. The QR payload is JSON:

```json
{
  "host": "127.0.0.1",
  "port": 17893,
  "token": "<bearer token>",
  "cert_fingerprint": null,
  "transports": ["loopback"]
}
```

### `POST /v1/pair`

Registers a client label for future revocation support.

Request:

```json
{
  "deviceLabel": "iPhone-15",
  "pushSubscription": null
}
```

Response:

```json
{
  "protocol_version": "1.0",
  "deviceId": "01J...",
  "machineId": "01J..."
}
```

### `WS /v1/realtime`

The WebSocket handshake uses the same bearer token. Native clients send it in `Authorization`. Browser clients running inside the trusted desktop app may use `?token=<token>` because browser WebSocket APIs cannot set arbitrary headers.

Server messages are JSON text frames:

```json
{"type":"approval-pending","protocol_version":"1.0","approval_id":"01J...","machine_id":"01J...","session_id":"01J...","agent":"claude-code","tool":"Bash","input":{},"cwd":"/repo","metadata":{}}
{"type":"approval-resolved","protocol_version":"1.0","approval_id":"01J...","machine_id":"01J...","decision":"allow","by":"desktop","reason":null}
{"type":"run-event","protocol_version":"1.0","machine_id":"01J...","session_id":"01J...","kind":"completed","payload":{"exit_code":0}}
{"type":"pty-output","protocol_version":"1.0","machine_id":"01J...","session_id":"01J...","data":"SGVsbG8K"}
{"type":"ping","protocol_version":"1.0","machine_id":"01J..."}
```

Clients may respond with `{"type":"pong"}`. Servers should drop clients that do not respond for 90 seconds once strict heartbeat enforcement is enabled.

## 4. Adapter Semantics

Claude Code uses an HTTP `PreToolUse` hook at `/v1/adapters/claude-code/hook`. Onibi transforms Claude's hook payload into `/v1/approval/request` and returns a Claude-compatible permission decision. Edited approvals include `updatedInput`.

Codex uses `onibi _hook codex` as a command hook. Phase-03 intercepts Bash tool calls only and emits `{"permissionDecision":"allow"}` or `{"permissionDecision":"deny"}` to stdout.

OpenCode, Gemini, Aider, Cursor, and Goose are registered as stubs in v1.0.

## 5. Persistence

SQLite lives at `~/.config/onibi/onibi.db`. The database stores approvals, run events, paired devices, and local metadata. Undecided approvals survive restart and are rebroadcast to WebSocket clients when they reconnect.

The bearer token is stored in the platform keyring when available. Headless systems without a usable keyring fall back to `~/.config/onibi/token.txt` with mode `0600`.

## 6. Failure Modes

- Missing or invalid bearer token: `401 Unauthorized`.
- Explicit protocol mismatch: `426 Upgrade Required`.
- Approval wait timeout: adapter receives `decision: "deny"` with `reason: "timeout"`.
- No clients connected: approval remains pending in SQLite until a client attaches or the request times out.
- Competing decisions: first decision wins; later calls return `409 Conflict`.
- Server restart: pending approvals reload from SQLite and are emitted over `/v1/realtime`.
- Malformed JSON: `400 Bad Request`.
- Internal storage or adapter errors: `500 Internal Server Error` with an error envelope.

## 7. Compatibility Rules

Clients and adapters must include `protocol_version: "1.0"` on envelopes they originate. Servers may accept missing versions from built-in adapters for compatibility, but must reject explicit non-`1.0` values.

Unknown fields must be ignored. Unknown WebSocket message types should be logged and skipped. Unknown run-event kinds should be displayed or persisted as opaque events.
